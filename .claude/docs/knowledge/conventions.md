# Naming Conventions — Single Source of Truth

## Requirement ID Pattern (R-PN-NN)

**Canonical regex**: `R-P\d+-\d{2}`

**Format**: `R-P{phase}-{sequence}` where:

- `R` = Requirement prefix (constant)
- `P{phase}` = Phase number (1-based integer, no zero-padding)
- `{sequence}` = Two-digit zero-padded sequence within the phase (01-99)

**Examples**: R-P1-01, R-P1-08, R-P3-12, R-P6-09

**Used by**:

- `PLAN.md` Done When sections — defines each requirement
- `prd.json` acceptanceCriteria[].id — links stories to requirements
- Test files — `# Tests R-PN-NN` marker in docstrings for traceability
- `verification-log.jsonl` — structured JSONL entries with `criteria_verified` and `overall_result`
- `/audit` Sections 3 and 8 — greps for markers to validate traceability chain

## Traceability Chain

```
PLAN.md "R-P1-01: [criterion]"
    ↓ (extracted by /ralph-plan Step 7)
prd.json { "id": "R-P1-01", "criterion": "...", "testType": "..." }
    ↓ (enforced by Builder TDD)
test file: # Tests R-P1-01
    ↓ (validated by QA Step 11)
verification-log.jsonl: {"story_id": "STORY-001", "criteria_verified": ["R-P1-01"], "overall_result": "PASS"}
    ↓ (audited by /audit Section 3)
Full traceability: requirement → story → test → verification
```

## Story ID Pattern

**Format**: `STORY-{sequence:03d}` (e.g., STORY-001, STORY-006)

Auto-generated by `/ralph-plan` Step 7 from PLAN.md phase numbers.

## Branch Naming

**Ralph feature branches**: derived per-worktree by `derive_ralph_branch_name()` in `_lib.py`.

- **Canonical repo root**: `ralph/[plan-slug]` (e.g., `ralph/v-model-upgrade`)
- **Linked human worktree** (`git worktree add`): `ralph/[plan-slug]-[8hex]` (e.g., `ralph/v-model-upgrade-a1b2c3d4`)

The 8-character hex suffix is a deterministic SHA-256 prefix of the normalized worktree path. This prevents two concurrent Ralph sprints in different linked worktrees from colliding on the same feature branch. Canonical roots remain unchanged for backward compatibility. The slug itself is derived from the `# Plan:` H1 in `.claude/docs/PLAN.md`, lowercased and hyphenated, with `"sprint"` as the fallback when no header is present. Run `python .claude/hooks/ralph_branch_name.py` to print the resolved name from any working directory.

## Commit Message Convention

**Format**: Conventional Commits (`feat:`, `fix:`, `docs:`, `chore:`)

**Ralph story commits**: `feat({story.id}): {story.description}`

Example: `feat(STORY-001): prd.json v2 schema template`

## Phase Type Convention

**Field**: `phase_type` in prd.json story objects

**Valid values**:

| Value         | Description                                     | QA Behavior                                |
| ------------- | ----------------------------------------------- | ------------------------------------------ |
| `foundation`  | Core infrastructure, utilities, base modules    | Skips integration tests (Step 4)           |
| `module`      | Self-contained feature modules                  | Runs all steps except E2E                  |
| `integration` | Cross-module wiring, API boundaries             | All steps run; integration tests mandatory |
| `e2e`         | Full system paths, deployment, end-to-end flows | All steps run; most rigorous               |

**Used by**:

- `prd.json` — each story can specify a `phase_type` for adaptive QA
- `qa_runner.py` — `--phase-type` argument controls which QA steps are skipped
- `/ralph-plan` Step 7 — auto-assigns phase_type based on plan phase characteristics

**Default**: If not specified, all QA steps run (equivalent to `e2e`).

## Frontend Verification

**Purpose**: Ensure frontend file changes are validated in a browser before story promotion.

**Trigger**: Applied when `has_frontend_files()` returns `True` for the story's changed files.

**`FRONTEND_EXTENSIONS` constant** (defined in `_qa_lib.py`):
The set of file extensions considered "frontend". Current values:
`.html`, `.css`, `.js`, `.ts`, `.jsx`, `.tsx`, `.vue`, `.svelte`

**`frontend_verified` field** (in `PromotionInputs` and `RALPH_WORKER_RESULT`):
- `True` — browser verification was performed and the UI renders correctly
- `False` — browser verification was not performed (or failed)
- `None` — not set; treated as `False` by the promotion gate

**`browser_evidence` field** (in `RALPH_WORKER_RESULT`):
- Optional string: URL, screenshot path, or description of what was verified
- Empty string if no evidence was captured

**Configuration** (`workflow.json` `ralph` section):

| Key                          | Values                          | Default  | Effect                                                                                                              |
| ---------------------------- | ------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------- |
| `ralph.frontend_verification` | `"warn"`, `"fail"`, `"disabled"` | `"warn"` | Controls `evaluate_promotion` Check 3d severity. `"warn"` allows promotion with a warning; `"fail"` blocks it. |

**Enforcement**: `evaluate_promotion()` Check 3d in `_qa_lib.py`. Reads `changed_files` from the QA receipt (written by `_write_receipt()` in `qa_runner.py`) and calls `has_frontend_files()` to determine if verification is required.
