## State Machine Validation

The load state machine enforces 8 canonical statuses:

- **draft** -> planned, cancelled
- **planned** -> dispatched, cancelled
- **dispatched** -> in_transit
- **in_transit** -> arrived
- **arrived** -> delivered
- **delivered** -> completed
- **completed** -> (terminal)
- **cancelled** -> (terminal)

All 8 valid forward transitions and 2 terminal state rejections are documented.
Invalid transitions return HTTP 422 (BusinessRuleError).

## Persistence Verification Patterns

Every mutation test in load-lifecycle.spec.ts includes a subsequent GET
to verify the change persisted to the MySQL database:

1. **verify created**: POST /api/loads -> GET /api/loads confirms load appears
2. **verify updated**: PATCH status -> GET /api/loads confirms status persisted
3. **persistence**: load_number and status survive a full reload of the list
