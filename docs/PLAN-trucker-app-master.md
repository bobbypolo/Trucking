## V-Model guarantee

Every sprint B1 → M traverses the full V: requirements (R-markers) →
design (file inventory + architecture docs) → implementation (Ralph
stories with 4-checkpoint TDD) → unit tests → integration tests (sprint
targeted verification) → system tests (`/verify` + `/audit`) →
acceptance tests (release checklist, operator-signed).

**R-markers are Ralph-automatable only.** Real-device validation,
simulator validation, EAS build invocation, store submission, legal
review, and any other operator-run validation are tracked as **release
checklist rows** signed off by a human operator, NOT as Ralph R-markers.

## SaaS regression protection strategy (restated every sprint that touches shared backend)

Every sprint that touches `server/`, `shared/contracts/`, or root
`package.json` runs the SaaS non-regression gate as part of its
targeted verification:

1. **Additive backend changes only** — existing routes/services/migrations
   are FROZEN unless explicitly additively extended (new fields, new
   query params — never renames, never removals).
2. **No web UI replacement** — web components are NOT touched except
   for the three explicit web additions scheduled: D (invitation panel),
   G (ELD admin), K (broker credit watchlist).
3. **Role/tenant contract tests** — every new route runs the existing
   `requireAuth` + `requireTenant` middleware tests to confirm the auth
   boundary is still enforced.
4. **Existing route regression** — any sprint that touches a route file
   runs the full existing test file for that route (not just its new
   tests) to catch regression.

## Locked decisions (final)

1. **Package manager**: root stays npm. `apps/trucker/` is an isolated
   npm subproject with its own `package.json` + `package-lock.json` +
   `node_modules/`. NO `pnpm-workspace.yaml`. NO root `"workspaces"`
   field. Independent lockfiles. Use `npm ci` (not `npm install`) in
   all verification and CI commands.

2. **Peer layout** — `apps/trucker/` is a peer directory to root
   `src/`, `components/`, `services/`. Web app stays exactly where it
   is. `shared/contracts/` gets additive-only extensions (new
   `mobile-*.ts` files + additive barrel edits).

3. **Phase 1 gap fix is additive** — Sprint A's `tax_year`,
   `packet_hash`, DB-blob storage is canonical. B1 adds migration
   with `aging_bucket` column, updates job, wraps in external scheduler.
   Bucket contract: `current=0`, `1_30=1..30`, `31_60=31..60`,
   `61_90=61..90`, `90_plus=>90`.

4. **Windows-safe tooling only** — All operator-run scripts are
   `.cjs`/`.mjs` (Node.js) or `.ps1` (PowerShell). NEVER `.sh` for
   anything the operator runs locally. All verification commands use
   cross-platform tools: `npx vitest run`, `npx tsc --noEmit`,
   `node scripts/*.cjs`. NEVER use `grep`, `wc`, `/dev/null`, `/tmp`,
   or shell pipelines. File counts and content checks go through Node
   scripts that use `fs` and `path`. **All R-marker assertions that
   read a file use `fs.readFileSync` + regex or AST walk — the word
   "grep" does not appear in any R-marker description.**

5. **Sprint B split into B1 + B2.** B1 = accounting/telemetry +
   program docs foundation + feature flags DB. B2 = peer layout +
   Expo scaffold + CI + directory lock + shared contracts extension
   + architecture doc first-pass content. B2 dispatches only after
   B1 merged.

6. **Native/store preflight starts in B2.** Bundle IDs + EAS project
   ID + minimum permission set in B2. Additional permissions added
   per-sprint as the feature that needs them ships. No speculative
   permissions.

7. **JIT external service setup.** Set up only when next sprint
   requires. Current state:
   - **WIRED**: Stripe, Twilio, Firebase Storage, jsPDF, jszip,
     `shared/contracts/`, `server/middleware/idempotency.ts` (with
     migration `004_idempotency_keys.sql`)
   - **PARTIAL**: Sentry (server-side init missing — fixed B1)
   - **NOT_STARTED**: PostHog (L), Motive (F sandbox, G live),
     Expo account (B2 for EAS project init), Apple Developer (M),
     Google Play Console (M), i18next (C)

8. **Two-layer completion**: Engineering complete (code + targeted
   tests + SaaS non-regression) vs Launch complete (+ real device
   validation + legal review + store submission).

9. **CI gates are per-sprint targeted**, not global. Baseline debt
   register absorbs pre-existing failures.

10. **Rolling handoff keyed on branch + merge SHA**, via Node.js
    `.cjs` script.

11. **SaaS non-regression gate runs every shared-backend sprint**
    (every sprint except M).

12. **Operator gates separated from Ralph R-markers.** Simulator
    validation, real-device testing, EAS builds, store submissions,
    legal review are release checklist rows signed by operator,
    NEVER Ralph R-markers.

13. **Migration-number management**: placeholders `<NEXT>` in plan,
    actual numbers assigned at dispatch via scanning
    `server/migrations/`.

14. **Feature flags consistent from B1**: `feature_flags` table
    migration + minimal read endpoint ship in B1 (so flags are
    DB-or-env from the first mobile sprint, not env-only for B2–G).

15. **Mobile domain layering pattern (mandatory)**: every mobile
    domain sprint adds all 4 layers:
    - `apps/trucker/src/services/<domain>.service.ts` — API call wrapper
    - `apps/trucker/src/hooks/use<Domain>.ts` — React Query hook
    - `apps/trucker/src/components/<domain>/*.tsx` — UI components
    - `apps/trucker/app/(app)/<screen>.tsx` — Expo Router screen entry
    Stories that add a screen without the service layer FAIL the
    plan compliance check.

16. **Every new server route story includes**: route file + mount
    change in `server/index.ts` + route test + **explicit auth-negative
    R-marker** (unauthenticated call → HTTP 401 or 403). Stories
    missing any of these FAIL the plan compliance check.

17. **Gate document ownership**: each architecture doc is written by
    the PREVIOUS sprint (never by the sprint that gates on it). The
    one exception is F's ELD provider contract which F Story-01
    writes as mandatory first story (no F dispatch gate on it; G
    gates on F merged). This exception is documented in Risk #1.

18. **Idempotency key format uses the EXISTING server middleware
    format**: `{actorId}:{endpoint}:{entityId}:{nonce}` (four
    colon-separated parts), matching
    `server/middleware/idempotency.ts#parseIdempotencyKey` (already
    shipped, backed by migration `004_idempotency_keys.sql`). Mobile
    clients construct keys in this format. The previously-considered
    `base64(userId-localId-createdAt)` format is REJECTED because it
    would require server-side parser changes.

#### `docs/trucker-app-auth-architecture.md` (B2, extended in D)

Required content:
- **Session model**:
  - Access token: JWT, 15-minute TTL, in-memory only (never persisted)
  - Refresh token: opaque random string, 30-day TTL, `expo-secure-store`
    (platform keychain encrypted)
  - Rolling rotation: every refresh call issues new refresh token +
    invalidates old one
- **Device ID**: UUID v4 generated on first launch, stored in
  `expo-secure-store`, sent as `X-Device-Id` header
- **Invite token**: JWT signed with `INVITATION_TOKEN_SECRET`, 7-day
  TTL, single-use (DB-tracked), branded link `https://disbatch.me/i/<token>`
- **Biometric unlock**: `expo-local-authentication`. Gates retrieval
  of refresh token from `expo-secure-store`. NOT a credential. 3
  failures → password fallback.
- **Offline re-auth**: if access token expired AND refresh token still
  within 30d → show cached data + offline banner → queue writes via
  `document-queue.ts` (reuses Sprint F plumbing) → refresh on reconnect
  BEFORE replay
- **Force logout**: dispatcher UI → `DELETE /api/user/sessions/:sessionId`
  → refresh token invalidated server-side → mobile app next refresh
  fails → hard logout
- **Owner-op recovery**: self-serve email reset token (7d TTL, uses
  existing password reset infra if present)

#### `docs/trucker-app-offline-sync-architecture.md` (E)

Required content:
- **Queue record zod schema**:
  ```ts
  z.object({
    localId: z.string().uuid(),
    userId: z.string(),
    tenantId: z.string(),
    operation: z.enum(['create', 'update', 'delete', 'upload']),
    resource: z.string(),
    resourceId: z.string().nullable(),
    payload: z.record(z.unknown()),
    attachments: z.array(z.object({ uri: z.string(), mimeType: z.string() })),
    retryCount: z.number().int().min(0).max(5),
    status: z.enum(['pending', 'uploading', 'failed', 'conflict', 'done']),
    idempotencyKey: z.string(),
    createdAt: z.string(),
    lastAttemptAt: z.string().nullable(),
    error: z.string().nullable(),
  })
  ```
- **Retry schedule**: `[1s, 4s, 15s, 60s, 300s]`, max 5 attempts, then `failed`
- **Idempotency key format**: `{actorId}:{endpoint}:{entityId}:{nonce}`
  (four colon-separated parts, matching existing server middleware
  `server/middleware/idempotency.ts#parseIdempotencyKey`). Mobile
  constructs: `actorId=userId`, `endpoint=POST:/api/documents`,
  `entityId=loadId-or-localId`, `nonce=uuid v4`.
- **Attachment lifecycle**: (1) enqueue → store blob URI, (2) uploading
  → request signed URL, upload in chunks, commit, (3) done → cleanup
  local blob after 7 days
- **Conflict policy**:
  - Default: LWW (last-write-wins by HLC timestamp)
  - Financial fields (settlements, expenses amount): field-level
    manual-resolve
- **Replay ordering**: HLC `{physical, logical, nodeId}` total order
- **Failure UX**: pending (spinner), uploading (progress bar),
  conflict (banner with Resolve CTA), failed (Retry button + error
  detail modal), done (checkmark, auto-dismiss 3s)

#### `docs/trucker-app-billing-policy.md` (I final story)

Required content — **must contain a real decision, not "TBD"**:

- **Decision**: Stripe web checkout only. NO StoreKit. NO Google Play Billing.
- **Rationale** (Apple App Store Guideline 3.1.3(b)):
  - LoadPilot is a B2B SaaS tool where drivers and fleet admins access
    subscription services consumed outside the app (web dispatch,
    accounting, compliance filing).
  - Guideline 3.1.3(b) exempts "enterprise services, business services,
    or educational services" from IAP requirement when the service is
    purchased and used outside the app.
  - Our mobile app is an access surface, not the primary purchase
    funnel; the primary purchase funnel is the web app.
- **Owner-op self-serve flow**:
  - Mobile "Upgrade" button → `Linking.openURL(<Stripe Checkout URL>)`
    → system browser
  - User completes checkout → Stripe webhook fires → tenant tier
    updated server-side → mobile app polls `/api/subscription/status`
    on resume → UI updates
  - NO in-app purchase, NO StoreKit init
- **Fleet upgrade**: web-only. Mobile shows "Upgrade at loadpilot.app"
  with external link.
- **Fallback plan if Apple rejects 3.1.3(b) exemption**:
  - Remove upgrade button from mobile entirely
  - Mobile shows "Manage your subscription at loadpilot.app" link
  - No functionality loss — free tier users simply cannot upgrade in-app
- **Legal review status**: requires sign-off before Sprint J merge.
  Track in release checklist.

## Mobile domain layering pattern (mandatory)

Every mobile domain sprint adds ALL 4 layers for each new domain.
Stories adding only a screen without service/hook FAIL the plan
compliance gate.

```
Service layer:     apps/trucker/src/services/<domain>.service.ts
                   (wraps shared/contracts DTOs, calls apiClient)

Hook layer:        apps/trucker/src/hooks/use<Domain>.ts
                   (React Query wrapper around service)

Component layer:   apps/trucker/src/components/<domain>/<Component>.tsx
                   (UI primitives scoped to domain)

Screen layer:      apps/trucker/app/(app)/<screen>.tsx
                   (Expo Router entry — composes hooks + components)
```

Example for Sprint E trip domain:
- `src/services/trip.service.ts` — `getCurrentTrip()`, `getTripHistory()`,
  `updateStatus()`
- `src/hooks/useCurrentTrip.ts` — React Query hook using trip.service
- `src/components/trip/ActiveTripCard.tsx`, `StopSequence.tsx`, etc.
- `app/(app)/trip.tsx` — composes the above

## Operator gates vs Ralph R-markers

**Ralph R-markers** are ALL automated and run by Ralph workers during
stories. They:
- Pass or fail automatically via test execution
- Never require human judgment
- Never require a physical device or simulator
- Never require an external account action
- Never require visual inspection

**Operator gates** are human-owned and tracked in
`docs/trucker-app-release-checklist.md`. They:
- Require operator signature to close
- Include simulator validation, real-device testing, EAS builds,
  store submissions, legal review, external account setup
- Do NOT block Ralph engineering-complete for a sprint
- DO block Launch complete for the program

### Operator gate types referenced by sprints

| Gate ID | Description | Owned by |
|---|---|---|
| `OP-ACCT-<X>` | External account setup (Twilio verify, Motive sandbox, Apple Dev, etc.) | Operator |
| `OP-SIM-<X>` | Simulator validation (iOS Simulator / Android Emulator) | Operator |
| `OP-DEV-<X>` | Real device validation (physical iPhone / Android) | Operator |
| `OP-EAS-<X>` | EAS build invocation (dev, preview, production) | Operator |
| `OP-STORE-<X>` | Store submission (TestFlight, Internal Track, production) | Operator |
| `OP-LEGAL-<X>` | Legal review (privacy policy, ToS, disclaimers) | Legal + Operator |

## V-Model Lifecycle Per Sprint

Every sprint B1→M traverses the full V. Sprint-level contract:

**Left arm** — R-markers (requirements) → file inventory + arch docs
(design) → Ralph stories with 4-checkpoint TDD (implementation)

**Right arm** — per-R-marker unit tests (unit) → sprint targeted
verification (integration) → `/verify` + `/audit` (system) →
release checklist operator gates (acceptance)

**Ralph dispatch invariants** (every sprint):
1. Worktree isolation per story worker
2. Checkpoint hash before each story
3. Feature branch `ralph/trucker-app-sprint-<x>`
4. 4-checkpoint TDD (Red → Green → Refactor → Gate)
5. Selective staging (no `git add -A`)
6. Format before commit
7. Fixture validation (collect-only)
8. Circuit breaker (3 consecutive skips → halt)
9. `needs_verify` cleared before next sprint dispatch
10. `npm ci` (not `npm install`) in all verifications
11. All R-marker assertions use `fs.readFileSync` + regex (no shell `grep`)

## Rolling Handoff Workflow

`scripts/sprint-handoff.cjs` (Node.js CommonJS, cross-platform):
```
node scripts/sprint-handoff.cjs <current-branch> <next-sprint-id>
Steps:
  1. git fetch origin (via child_process.execSync)
  2. git merge-base --is-ancestor origin/<current-branch> origin/main
     - exits 0 if merged; halts if not
  3. Capture merge SHA via git log (via execSync)
  4. Append to docs/trucker-app-sprint-history.md
  5. Archive .claude/docs/PLAN.md → docs/PLAN-trucker-app-sprint-<X>-executed.md
  6. Extract sprint-<Y> section from docs/PLAN-trucker-app-master.md
  7. Write extracted content to .claude/docs/PLAN.md
  8. Regenerate .claude/prd.json via /plan generator
  9. Reset .claude/.workflow-state.json ralph block
 10. Print "READY: git checkout -b ralph/trucker-app-sprint-<Y> && /ralph"
```

All filesystem operations via `fs`, all git via `child_process.execSync`.
No shell pipes, no `grep`, no `wc`, no `/tmp`.
